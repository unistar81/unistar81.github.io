WebPlex Design System

About

The Design Language System is the standardization of all user interface and brand components across WebPlex. These elements have been abstracted from the core WebPlex code base so that they can be updated quickly and globally. This means that there will be no "special" or one-off element designs and that all colors are managed in a controlled way.

The design Language System features consistent, descriptive and semantically correct markup that can be reused across templates and components.

The configuration relies on CSS4 Variable config that is used to customize and style multiple brands and templates with ease, providing a unique look and experience to every brand. The CSS structure and implementation removes a need for a post CSS processing for customization. It is framework independent and can be deployed to any framework with ease.

The system allows for fast deployment, modification and changes across a global infrastructure.

UX Principles

Design does not occur in a vacuum. This is especially true in large companies where many different teams and departments are involved in turning design into a reality. Establishing design principles and aesthetic guidelines allows teams to start building products immediately, with limited intervention, which ultimately makes our sprints more efficient. Teams can also build with confidence that new products will fit into a larger vision for Content Platform.

Clarity

It is our responsibility to create products that are user-centric. These are products and experiences that are easy to understand by the primary users. Clarity in visual design is achieved by adhering to the principles balance, hierarchy and proximity. Clarity in language consists of short, descriptive sentences.

Consistency

We want users to feel familiarity across different products in Content Platform family so that they can spend their time more efficiently, instead of figuring out how the tool works. Users may be switching back and forth from different products in the same suite, so it important to be consistent in our use of navigation, UI elements and expected paradigms.

Efficiency

Our goal is to reuse design elements throughout different products. Reusing components, instead of designing them from scratch each time, will hopefully reduce iteration periods. We hope that these guidelines will be a self-service tools for engineers, so that they can start building their projects knowing that the design elements have been approved and agreed on.

How to Use a Design System

Building software requires flexibility, collaboration and cooperation. We hope that these guidelines will help teams save time and energy when challenged with design decisions.

InVision DSM

The WebPlex Design System is the documentation of our UX and UI principles. The colors, components, etc. can be used in Sketch through the InVision DSM tool.

Theming

The DLS uses a configurable theme object that allow us to deliver a theme by brand, language, and region. The theme variables are merged into the configuration object, each inheriting or overriding the values from its parent the hierarchy. This means that only the variables that need to be overridden at a higher level should be included in the higher level theme config. The Base CSS is then generated and is included in the page via middleware route.

To view the different brand themes, select a brand from the dropdown available on every page.

Using Typefaces in WebPlex

For each brand Weplex supports up to two font families: a primary and secondary:

Primary Font Family

Secondary Font Family

If a brand uses a singular font, the primary and secondary are the same. It is strongly recommended that all font families have multiple weights with italic versions:

Normal 400

Italic 400

Bold 700

Bold Italic 700

Body Copy

Weplex uses two sizes for body copy. The smaller size or p2 is used for maximizing screen real estate in smaller areas such as content descriptions while the larger size or p1 is used for long-form reading such as article text.

Overview

The WebPlex Design Language System is intended to create standardized HTML markup and base CSS classes using Atomic Design methodology to provide a general, reusable structure for our markup within each component across our apps. Class modifiers are then created using CSS and are applied to the parent containers to override how the markup is displayed to the end user.

Atomic Design Methodology

WebPlex utilizes the idea of Atomic Design Methodology which breaks components into individual resources that can be reused to create larger structures.



Pages

Each Page consists of the following elements which are used to compose the user interface:

  • Templates: Consist of one or more Organisms and can be defined by template class modifiers to determine how components display within the page template

  • Header and Footer: Unique Organisms that don't follow the structure of a LineList and are comprised of reusable Molecules and Atoms

  • LineList Components: A Page is composed of LineList Organisms, each of which consists of one or more Card Organisms. Each LineList receives one or more Class Modifiers which defines how the content will display

  • Cards: An iterating group of items within a component comprised of Molecules and Atoms. Each receives a data-driven "type" class which defines how the content within the item type will display

  • Meta and Media: Each card can have Meta and Media Molecules that are comprised of data-driven Atoms. For instance, if meta information or media information are not defined, they will not render. The same is true for Atoms within the meta or media components.

Templates

Page templates are containers to hold multiple module containers. They can also contain class modifiers which define the display of modules within the page template.
All modules within a page template are "LineList" components, with the exception of the Header, Footer, and Bala Notifier components.



LineList Component (Organism)

The LineList Component consists of multiple data-driven components and it's display is defined by class modifiers that are appended to the section node.

The main components consist of:

  • Module Container with modifier classes to define how the component itself will display

  • Section Header (data-driven) that will display the title of the component

  • Filter (data-driven) that will display a filter for the component

  • Grid Container containing the iterative Cards

  • Cards receive a {type} class in order to:

  • Define how content within the item type will display

  • Allow for a mixed grid container of items

  • Load More (data-driven) component to control component pagination

Card Components (Organism)

Single items within a Grid Container are iterative and are also composed of standardized, data-driven components. Display of items are controlled using a "type" class applied to each item which is data-driven.

A Card consists of the following data-driven components:

  • Media Component

  • Meta Component


Media Component (Molecule)

Within a Card Component, the Media Component consists of an Image or Video (depending on the data provided), as well as different labels and timestamps that are rendered if the data exists.

  • Picture or Video

  • Alert Label (e.g., Video, Episode)

  • Auth Label (e.g., Sign in to watch)

  • Timestamp Label (e.g., 20:13)


Meta Component (Molecule)(Organism)

Within a card component, the Meta Component consists of test data that describes the Media and calls to actions such as buttons and links. These include:

  • Header
  • Sub-header
  • Deck
  • Meta info
  • Buttons

CSS Config

Much like SASS and LESS, CSS4 added a variables feature which is supported by all major browsers. Rather than use a CSS framework, WebPlex uses the built-in CSS4 variables.

  • Brand-specific properties of the CSS classes are mapped to pre-defined CSS variables as designated by the design team
  • CSS4 variables are generated from theme.default.js in each brand config via middleware (see Generated CSS documentation)
  • Please refer to the Design Team documentation for specific variable declarations

Webplex supports the use of a Sponsor Logo Asset (SLA) within our templates. SLA’s can be used to sponsor pages, modules (content packages) and individual promo cards.

What’s New: SVGs

SLA’s should be submitted as a Scalable Vector Graphics or SVGs. This allows for a number of improvements:

  • Logo sizes are defined by the objects that house them
  • If logo placement sizes change no update needed for SLA template
  • Clear viewability on devices with retina displays
  • Generally smaller file sizes

Logo Requirements

When creating a SLA you first need the sponsor’s logo with the following requirements:

  • The original file must be in a vector format (.ai or .eps)
  • Although it can be color, logos must be a version for use on dark backgrounds according to the sponsor’s brand guidelines
  • Typography must be converted to outlines
  • Artboards must be the same size as logo

Extra Features

Page Background

WebPlex can support many types of backgrounds other than the default UI color.

Gradients

Gradients can be applied to the page background by modifying the --page-background variable with a gradient

Images

Images can be added to the page background by modifying the --page-background variable with an image path. The image can be in any format, and can also be used with a gradient as well.
